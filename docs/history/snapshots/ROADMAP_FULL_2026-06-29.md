## 3. v0.2 多人物与站位

> 状态：工程已落地（Claude Code v0.2.0，2026-06-18）；视觉规则已定稿（Codex v0.2.1，2026-06-18）；A/B 切换黑屏、侧视被侧墙遮挡、重复拖拽控制器等稳定性问题已修到 v0.2.3。加/复制/删人物、逐人选择/编辑/拖拽、多人投影、A/B 含多人均通过本地验收。默认多人站位与命名/颜色规则见 `RENDERING_SPEC.md §6`。

### 目标

让导演能快速模拟多人站位、遮挡、投影和灯光关系。

### MVP 范围

- 添加人物。
- 删除人物。
- 复制人物。
- 选择人物。
- 重命名人物。
- 调整人物位置。
- 调整人物朝向。
- 调整人物身高。
- 调整人物肤色和服装颜色。
- 最多先支持 5 个人，默认建议 3 个人以内。
- 多人物均能投射阴影。
- 多人物之间可以互相遮挡。
- 俯视图显示每个人的位置和朝向。
- A/B 对比中包含多人物状态。

### 暂时不做

- 群演级大数量人物。
- 路径动画。
- 碰撞约束和自动避让。
- 精确人物身材系统。

## 4. v0.3 白棚结构、道具和人台

> 状态：v0.3.4 已落地（2026-06-18）。Claude Code 已完成结构/道具工程接入；Codex 补齐“直播圆形小舞台”、人物离地高度/承载面放置，以及灯光 `manual/person/peopleCenter` 目标模式。下一步应进入 v0.4 人物基础姿态，或先让 Claude 做一轮 v0.3 交互打磨。

### 目标

让 Direct Light 不只模拟“人在空白棚里被灯照”，也能模拟拍摄现场常见的基础结构。这样导演、摄影、美术、制片在拍摄前可以沟通：

- 人和桌椅的相对位置。
- 灯打到桌面、椅背、人台、背景板后的阴影。
- 产品台、台座、椅子是否遮挡人物。
- 道具和结构会不会吃光、反光、投影到白棚。
- 镜头里实际看到的空间关系。

### 产品原则

这不是完整美术置景软件。第一版只做“拍摄沟通够用”的结构和道具：

- 尺寸清楚。
- 位置清楚。
- 材质大类清楚。
- 能投影、能挡光。
- 在俯视图里能看懂。

### v0.3c 服装人台与人物站台

这里要分清两个概念：

- 服装人台：mannequin，用于看服装、胸口受光和轮廓光。
- 人物站台：可承重平台，例如直播用圆形小舞台，用于把真人放高。

第一版服装人台建议：

- 半身人台。
- 全身人台。
- 服装展示人台。
- 可调高度。
- 可调朝向。
- 可调颜色：白、灰、黑、肤色、布面。
- 可投影和接收阴影。

第一版人物站台建议：

- 低矮长方形平台。
- 直播圆形小舞台，默认 `1.2m × 1.2m × 0.3m`。
- 人物可以放到台面上，脚底高度等于台面高度。

暂时不做：

- 人台复杂曲面精度。
- 可换衣服。
- 布料模拟。
- 真实品牌人台型号。

### v0.3d 现场沟通交互补齐

v0.3 不只增加物体，还要把“拍摄前沟通”需要的直接操作补齐。

人物交互：

- 人物继续支持直接拖动，拖动范围限制在白棚地面内。
- 人物支持 `position.y` 离地高度，第一版面板范围 `0..1.5m`。
- 人物可以一键放到桌面、椅面、凳面、沙发坐面、台座、箱体或直播圆形小舞台上。
- 在自由、俯视、侧视里都能选中人物；俯视是最主要的站位编辑视图。
- 右侧面板继续保留精确数值输入，用于和拖动互补。
- A/B 中改 A 的人物位置时，B 保持冻结。

灯光目标模式：

- 主人物：沿用当前 Actor A / `people[0]` 作为目标。
- 锁定人物：自动对准指定 Actor，人物移动或升高后目标随之更新。
- 多人中心：自动对准当前所有人物的目标点中心，适合群像或访谈。
- 对准一次：把当前灯目标写成某个人的位置，然后保持手动。
- 指定目标点：后续可在俯视图放一个 target marker，灯光对准该点。

白棚结构交互：

- 桌椅、台座、人台和背景板都应能拖动、旋转、复制和删除。
- 俯视图中结构对象需要显示外轮廓和朝向。
- 对象移动时实时更新遮挡、受光和投影。

### 目标

让人物不只是“站桩”，可以用基础姿态测试遮挡、轮廓光和身体阴影。

### 产品原则

Direct Light 不是角色动画软件。姿态功能服务于灯光判断：

- 手会不会挡脸。
- 头转向后脸部受光怎么变。
- 侧身后轮廓光是否明显。
- 身体前倾或转身后，影子形状是否变化。

### v0.4b 追加拆分：人物与承载物联动

这个功能承接用户关于“人物可以放到桌子、椅子、小舞台上，也可以上下移动”的需求。当前 v0.3.4 只完成了高度放置，v0.4b 要让姿态也跟着合理变化。

优先级：

1. ✅（v0.4.4）椅子 / 沙发 / 凳子：选择“放到承载物”时，把人物移动到坐面中心，并应用坐姿预设。
2. ✅（v0.4.4）小舞台 / 台座 / 箱体 / 桌面：人物脚底落到台面高度，保持站姿或用户当前姿态。
3. ✅（v0.4.5）拖动 / 移动 / 旋转物体时，如果人物已绑定在该承载物上，人物跟随移动；删除承载物、回到地面或手动挪人时解绑。
4. 拖动人物靠近承载物时，后续可做轻量吸附；第一版仍可通过下拉选择，避免误吸附。
5. ✅（v0.4.4）坐姿第一版不做完整 IK，只需要大腿/小腿/躯干形成可读的“坐着”，用于判断灯光遮挡和影子。

> 当前实现说明：v0.4.5 的“放到承载物”会建立实时绑定。人物保存承载物 id、本地 X/Z 偏移和相对朝向；`setObjectPositionXZ` / `updateObject` / `rotateObject` 会重算绑定人物的世界位置和朝向；“回到地面”、删除承载物或手动改人物位置会解除绑定。换姿态但位置不变不会误解绑。

已新增字段：

```ts
supportObjectId?: string;
supportLocalOffset?: { x: number; z: number };
supportRotationOffset?: number;
```

## 9. v0.8 更多光源 / 多灯管理

### 目标

开源之后先提高灯光上限，让 Direct Light 支持更复杂但仍然可读的布光沟通。

第一版开源保持默认 Key / Fill / Rim 三盏灯，是为了保证界面简单、性能稳定、代码容易读。现在核心功能已经稳定，下一版优先增强布光能力，而不是先抽多语言。

### MVP 范围

- 灯光上限从 3 提高到 6；是否提高到 8 留到性能和 UI 验收后再判断。
- 默认场景仍保持 Key / Fill / Rim 三盏，避免打开即显得复杂。
- 左侧灯光列表支持更清楚的多灯管理：重命名、复制、启用/停用、删除、满额状态。
- 当灯数较多时，列表需要保持可扫描，优先做折叠或紧凑摘要，不把 `LightsSection` 撑成大文件。
- 右侧灯光面板继续显示当前灯具、附件和有效光质；新增字段放在 `src/ui/light-panel/*` 小组件中。

### 为什么先做

- 多光源直接增强 Direct Light 的核心价值，比先做多语言更能扩大真实使用场景。
- 现在状态和组件结构仍简单，适合先把核心数据结构和列表交互定稳。
- 过早做 i18n 会让后续每个新增字段、按钮和空状态都同步维护多套文案，拖慢功能迭代。
- 多灯管理会决定后续自定义灯具预设、分组、A/B 摘要和导出说明的字段形态，应该先稳定。

### 数据与架构原则

- 继续让 `SceneConfig.lights` 作为灯光数组，不新增第二套 light store。
- 若需要灯光分组或折叠，优先使用轻量可选字段，旧方案经 `sceneMigration` 补默认值。
- 场景数据只存语义 ID 和数值，不存 UI 显示文案；例如继续存 `fixturePresetId` / `modifierId` / light `name`，不要把未来语言包文案写进场景数据。
- 多灯 UI 写在 `src/ui/object-list/*` 和 `src/ui/light-panel/*`，不要回填到 `ObjectList.tsx` / `LightPanel.tsx` 导出壳。
- 有效光质和灯光摘要继续走 `src/domain` 纯函数，渲染层只消费结果。

### 暂时不做

- 真实光度曲线或 IES 文件。
- 云端灯组库。
- 复杂灯组时间轴。
- 自动布光建议。
- 把默认场景改成 6 盏灯。

### 目标

让用户把自己常用的灯具参数保存成可复用的自定义灯具预设，并能导入/导出一份严格受控的 JSON。

这里说的“上传自定义参数”先收窄为“自定义灯具预设导入/导出”，不做任意场景参数上传，也不做真实 IES / 官方光度曲线解析。

### MVP 范围

- 用户可以把当前灯光参数保存为一个自定义灯具预设。
- 用户可以导出自定义预设 JSON。
- 用户可以导入自定义预设 JSON，并在器械下拉中看到它。
- 导入时做严格 schema 校验：字段、类型、范围、必填项都要检查；坏数据不能污染当前场景。
- 自定义预设只表达现有 `FixturePreset` 和 `LightConfig` 能承载的语义：类型、强度、光束角、柔硬、距离、颜色/色温能力、用途标签和备注。
- 自定义预设随本地保存保留，但不写入默认内置预设数组。
- A/B 可以比较“内置灯具预设 vs 自定义灯具预设”的画面差异。

### 暂时不做

- 任意完整场景 JSON 上传。
- IES、LDT、DMX 或厂商专有光度文件。
- 在线共享灯具库。
- 从网页 URL 抓取灯具参数。
- 自动判断品牌参数真伪。

### 数据与架构原则

- 内置预设继续在 `src/data/fixturePresets.ts`；用户自定义预设单独走本地存储或独立数据入口，不改坏内置数据。
- 导入格式必须版本化，例如 `schemaVersion: 1`。
- 导入 helper 放在 `src/domain` 或 `src/lib` 的纯函数中，UI 只显示校验结果。
- 场景中的灯仍只存 `fixturePresetId` 和实际灯光参数；如果引用的自定义预设缺失，场景也应能靠实际参数继续打开。
- 不把中文/英文显示文案写进场景数据；预设可以有用户自定义 `label`，但内置类别和能力标签仍用语义字段表达。

### 验收标准

- 导出后的 JSON 重新导入，预设数量和关键字段一致。
- 导入非法 JSON、缺字段、越界数值或未知枚举时，应用不崩溃，当前场景不被污染。
- 应用自定义预设后，行为与内置预设一致：只设置默认光质，之后仍可手动微调。
- 保存/加载、A/B、复制灯光和导出截图都保留应用后的灯光参数。

## 11. v0.10 多语言 UI

> 状态：✅ 已完成并通过用户真机视觉验收（Claude-led，2026-06-24）。唯一规格见 `V0_10_I18N_SPEC.md`，内置显示标签 / `sceneDiff` 细节见 `V0_10_1_DISPLAY_COPY_SPEC.md`。v0.10a 基础设施、v0.10b 核心 UI 抽取、v0.10.1 内置显示标签 + 派生文案、v0.10d 收口均完成；作为首个正式大版本 `v1.0.0` 发布，TopBar / package / Tauri / Cargo 版本元数据均为 `1.0.0`。

### 目标

在 v0.8 多灯管理和 v0.9 自定义灯具预设稳定后，加入运行时 UI 多语言。多语言只翻 UI 表达，不翻译场景内容、用户输入、保存方案或自定义器械数据。

首批语言：

- 简体中文。
- 英语。
- 日语。

### 实现原则

- 默认语言：简体中文。
- 语言切换放在设置或顶栏的轻量入口，不影响主操作。
- UI 文案、按钮、面板标题、空状态、导出说明都进入语言包。
- 灯具型号、品牌名、数值单位不翻译，只翻译类别和说明。
- 保存的方案数据不依赖语言，切换语言不改变场景内容。
- 场景数据继续只存语义 ID 和用户输入值，不存语言包输出后的显示文本。
- 不引入重型 i18n 依赖；优先使用项目内轻量 typed dictionary。
- 语言偏好只存在 `direct-light.language.v1`，不写入 `SceneConfig`、A/B 快照、保存方案或自定义器械 JSON。

### 写入范围建议

- 在 store 顶层新增 `language` + `setLanguage`，持久化到 `src/lib/storage.ts` 的独立 key。
- UI 组件逐步改为读取文案 key；内置 fixture / scene-object / modifier / camera preset 显示标签通过 id 映射本地化，原始数据表不改写。

## 13. v0.4.6 A/B 对比产品引导

> 状态：✅ 已通过 Codex 验收（Hermes 起草，Codex 复核修正，2026-06-19）。差异摘要、B 冻结时间、空状态三动作卡、A/B 文案更新都已接入。Codex 已修正一个流程问题：进入对比不再自动生成 B，因此空状态引导卡可以真实出现。

### 目标

让 A/B 从“一个可用视图”变成“导演沟通工具”：

- A 是正在编辑的当前方案，B 是冻结或加载的参考方案。
- 用户一眼知道 B 的来源：冻结当前、已存方案、交换前方案。
- 用户能快速开始一次对比，而不是先理解内部逻辑。
- 对比时能看到“这次主要在比什么”：灯位、人物位置、道具位置、姿态、摄影机或白棚。
- A/B 继续作为后续摄影机、灯具、控光附件的默认验收容器。

### MVP 范围

- 对比顶部状态更清楚：A 标为“当前编辑”，B 标为“冻结参考”。
- B 为空或刚进入对比时，给出明确的下一步动作：冻结当前为 B、选择已存方案、或回到普通视图继续调整。
- 冻结 B 后，显示 B 的时间/来源/名称，避免用户忘记 B 是什么。
- 交换 A/B 前后，文案要明确“交换后正在编辑哪一边”。
- 对比画面增加简短差异摘要：灯光、人物、道具、姿态、摄影机、白棚，先做大类提示，不追求完整 diff。
- 给 3 个导演常用对比入口：灯光前后、站位前后、道具/小舞台位置前后。

### 暂时不做

- 完整逐字段 diff 表。
- 多版本时间轴。
- 评论批注系统。
- 云端协作。
- 自动判断哪个方案更好。

### 设计原则

- 不写长教程；用清晰标签、按钮状态、空状态和短提示完成引导。
- A/B 的核心动作只保留少数几个：冻结、选择 B、交换、保存方案。
- 对比摘要服务于沟通，不服务于调试；只告诉导演“差异大概在哪类”。
- 所有引导文案都要短，不能遮挡画面。

### 验收标准

- 用户第一次进入对比视图时，不需要解释也能开始冻结 B。
- 冻结 B 以后，用户能说清楚“左边是我正在改的，右边是不动的参考”。
- 改灯位、人物位置或道具位置后，对比摘要能提示主要差异类型。
- 交换 A/B 后，用户不会误以为正在编辑 B。
- A/B 从对比视图切回其他视图不黑屏，已有稳定性不能回退。
